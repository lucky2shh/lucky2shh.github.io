Nacos系列-Nacos原理解析

Nacos Nacos原理
Nacos
发布日期:   2023-09-14
更新日期:   2023-08-07
文章字数:   2.6k

1、服务注册

1.1、客户端

- SPring启动时 加载 自动配置类 NacosServiceRegistryAutoConfiguration
- 注入 return new NacosAutoServiceRegistration(...)
	- 其父类 AbstractAutoServiceRegistration 也被初始化了
		- 实现了ApplicationListener接口，监听Spring容器启动过程中的事件
		- 监听到 WebServerInitializedEvent（web服务初始化完成）的事件后，开启服务注册流程。
			1、发布开始注册的事件
			2、开始注册：register();
			3、发布注册完成事件
			4、服务状态设置为运行状态。

- 开始注册：register();
- NacosServiceRegistry.register(Registration registration);
	`NacosServiceRegistry`是Spring的`ServiceRegistry`接口的实现类，
	而ServiceRegistry接口是服务注册、发现的规约接口，定义了register、
	deregister等方法的声明。
- NamingService.registerInstance(serviceId, group, instance);
	NamingService接口的默认实现就是NacosNamingService。
- NacosProxy.registerService(groupedServiceName, groupName, instance);
	发送注册服务实例请求。发送到 /nacos/v1/ns/instance

1.2、服务端

- 入口 com.alibaba.nacos.naming.controllers.InstanceController

- 将 实例信息 封装成 Instance对象。然后 执行注册实例方法。
	ServiceManager.registerInstance(namespaceId, serviceName, instance);
- 将 要注册的实例放入Service中
	addInstance(namespaceId, serviceName, instance.isEphemeral(), instance);
	
	- 对服务加synchronized（不同服务不会阻塞，同一个服务的不同实例会被阻塞）
		- 将实例列表封装成 instances 对象
		- 更新注册表，完成 Nacos集群数据同步。 consistencyService.put(key, instances);

- 更新服务列表 ServiceManager.updateIpAddresses(...);
	先获取旧的实例列表，然后把新的实例信息与旧的做对比，新的实例就添加，老的实例同步ID。
	然后返回最新的实例列表。

- 集群一致性 更新 
	- DelegateConsistencyServiceImpl.put(..);
		- 根据实例是否是临时实例，判断委托对象
			// 判断是否是临时实例：（默认情况下，所有实例都是临时实例，我们关注DistroConsistencyServiceImpl即可。）
			// 是，选择 ephemeralConsistencyService，也就是 DistroConsistencyServiceImpl类
			// 否，选择 persistentConsistencyService，也就是PersistentConsistencyServiceDelegateImpl

		- 先将要更新的实例信息写入本地实例列表，然后同步集群数据
			- onPut(key, value);
				- 将任务放入阻塞队列，通过线程池异步从队列中取出任务来执行。
					notifier.addTask(key, DataOperation.CHANGE);
					handle(pair);
				- 根据 CHANGE事件 还是 DELETE 事件，来执行对应的操作。
					- 获取旧实例列表
					- 检查新加入实例状态
					- 移除要删除的实例
					- 覆盖旧实例。
			- distroProtocol.sync(...);
				- 遍历集群所有节点（除了自己），交给线程池去执行。
					NacosDelayTaskExecuteEngine.addTask(distroKeyWithTarget, distroDelayTask);
					NacosDelayTaskExecuteEngine，这个类维护了一个线程池，并且接收任务，执行任务。
					去除任务来执行是基于Distro模式的同步是异步进行的，并且失败时会将任务重新入队，因此不保证强一致性，属于AP模式的一致性策略。

1.3、总结

（1）Nacos的注册表结构是什么样的？

Nacos是多级存储模型，最外层通过namespace来实现环境隔离，然后是group分组，分组下就是服务，一个服务可分不同的集群，集群中包含多个实例。因此其注册表结构是一个Map结构：

Map<String, Map<String, Service>>，key是namespace_id，value代表的map结构的key是group+serviceName.

Service内部维护一个Map结构：Map<String,Cluster>，key是clusterName，值是集群信息，Cluster内部维护一个Set集合，元素是Instance类型，代表集群中的多个实例。

（2）Nacos如何保证并发写的安全性？

注册实例时会对service加锁，不同service之间本身就不存在并发写问题，互不影响。相同service时通过锁来互斥。并且，在更新实例列表时，是基于异步的线程池来完成，且线程池的线程数量为1。

（3）Nacos如何避免并发读写的冲突？

Nacos在更新实例列表时，会采用 CopyOnWrite 技术，首先将Old实例列表拷贝一份，然后更新拷贝的实例列表，再用更新后的实例列表来覆盖旧的实例列表。

（4）Nacos如何应对阿里内部数十万服务的并发写请求？

Nacos内部会将服务注册的任务放入阻塞队列，采用线程池异步来完成实例更新，从而提高并发写能力。

2、服务心跳

2.1、客户端

- 入口 NacosNamingService.registerInstance(...);
	- 触发心跳：BeatReactor.addBeatInfo(groupedServiceName, beatInfo)
		- 利用线程池，定期执行心跳任务，周期为 beatInfo.getPeriod()，默认5S
			executorService.schedule(new BeatTask(beatInfo), beatInfo.getPeriod(), TimeUnit.MILLISECONDS);

			发送心跳，然后判断心跳结果，若失败则重新 注册实例。
- 发送心跳 NamingProxy.sendBeat(...)
	请求地址：/v1/ns/instance/beat

2.2、服务端

- 入口：InstanceController.beat();

- 从Nacos注册表中获取实例
- 实例为null，表示获取失败 说明心跳失败，实例未注册
	- 重新注册新实例
- 实例不为null，则从注册表中获取服务service
- 若service为null, 则服务不存在，抛出异常
- 如果心跳没有问题，就处理 心跳结果。

- 处理心跳请求
	- 从集群中找到当前的心跳实例
	- 更新 该实例 最后一次心跳时间 lastBeat。

2.2.1、异常检查

- 入口 Service.init(); 服务注册时会被调用。

- 开启心跳检测任务
	HealthCheckReactor.scheduleCheck(clientBeatCheckTask);
	该任务会 每5秒对实例心跳状态 做一次检测。

	- clientBeatCheckTask
		- 拿到 所有临时实例列表
		- 判断 心跳间隔（当前时间-最后一次心跳时间）是否大于 心跳超时时间，默认15S
			- 超时，则标记实例为 不健康。
			- 不超时，发布实例状态变事件。
		- 判断心跳间隔（当前时间 - 最后一次心跳时间）是否大于 实例被删除的最长超时时间，默认30秒
			- 超过30S，删除实例。

2.2.2、主动检查

- 入口 ServiceManager.registerInstance(...)

- 若服务不存在，创建新服务。
	- 尝试获取服务
	- 服务不存在，开始创建新服务。
	- 将服务写入注册表 并初始化。
		- 初始化逻辑
			- 开启心跳检测任务
			- 遍历注册表中的集群
			- 完成集群初始化
				- 创建健康检测任务
				- 开启对 非临时实例 的定时检测。
- 非临时实例 定时检测逻辑
	- 获取所有 非临时实例 集合
	- 将检测信息 封装成 beat
	- 放入阻塞队列中。
	
	- 拿出任务去执行
		- 将任务封装为一个 TaskProcessor，并放入集合。
		- 批量处理集合中的任务。
			- 获取实例信息
			- 基于NIO建立TCP连接
			- 注册 连接、读取 事件。

2.3、总结

Nacos的健康检测有两种模式：

  • 临时实例：
    • 采用客户端心跳检测模式，心跳周期5秒。
    • 心跳间隔超过15秒就标记为不健康。
    • 心跳间隔超过30秒会被从服务列表删除。
  • 永久实例：
    • 采用服务端主动健康检测方式。
    • 周期为2000 + 5000毫秒内的随机数。
    • 检测异常只会标记为不健康，不会删除。

为什么Nacos有临时和永久两种实例呢？

以淘宝为例，双十一大促期间，流量会比平常高出很多，此时服务肯定需要增加更多实例来应对高并发，而这些实例在双十一之后就无需继续使用了，采用临时实例比较合适。而对于服务的一些常备实例，则使用永久实例更合适。

与eureka相比，Nacos与Eureka在临时实例上都是基于心跳模式实现，差别不大，主要是心跳周期不同，eureka是30秒，Nacos是5秒。

另外，Nacos支持永久实例，而Eureka不支持，Eureka只提供了心跳模式的健康监测，而没有主动检测功能。

3、服务发现

3.1、客户端

- 入口 NacosNamingService. getAllInstances(...);

- 判断 是否需要 订阅服务（默认true）
	- 若要订阅，执行订阅逻辑。serviceInfo = hostReactor.getServiceInfo(...);
	- 否则，执行拉取逻辑。serviceInfo = hostReactor.getServiceInfoDirectlyFromServer()
- 从服务信息中获取实力列表并返回。

- 拉取服务
	- 查看本地服务列表缓存是否存在，缓存是map
	- 不存在
		- 创建空ServiceInfo
		- 放入缓存
		- 放入待更新的服务列表
		- 立即更新服务列表
			- 执行HostReactor中的updateService()方法。
				- 基于ServerProxy发起远程调用，查询服务列表
		- 从 待更新服务列表 中删除
	- 存在，但需要更新
		- 执行更新逻辑。等待5秒中，待更新完成
	- 开启定时更新服务列表的功能
		scheduleUpdateIfAbsent(serviceName, clusters);
	- 返回缓存中的服务信息
		serviceInfoMap.get(serviceObj.getKey());
	
- 推送服务
	基本思路是：
		- 通过PushReceiver监听服务端推送的变更数据
		- 解析数据后，通过NotifyCenter发布服务变更的事件
		- InstanceChangeNotifier监听变更事件，完成对服务列表的更新

	入口 PushReceiver
		- 创建 UDP客户端
		- 准备线程池
		- 开启线程任务，准备接收变更数据
			- 接收推送数据
			- 解析为json字符串
			- 反序列化为对象
			- 交给 HostReactor去处理
			- 发送ACK回执

	- 处理接收到的数据 HostReactor.processServiceJson(String json);
		- 解析 接收到的数据，解析出ServiceInfo
		- 查询缓存中的 ServiceInfo
		- 如果缓存存在，则需要校验哪些数据要更新
			- 将接收到的ServiceInfo 放入缓存。
			- 对缓存中的数据 与 接收到的数据 进行对比。拿到
				新增实例、待删除实例、要修改实例。
			- 发布实例状态修改事件。
			- 将接收到的数据ServiceInfo写入缓存。
		- 缓存不存在
			- 放入缓存
			- 直接发布状态修改事件。

3.2、服务端

- 入口 InstanceController.list();

- 获取客户端UDP端口
- 获取服务列表并返回
	- 将消费者UDP端口、IP等封装成一个PushClient对象，存储到PushService中，
		后期服务变更时，方便推送消息。pushService.addClient()
	- PushService实现了ApplicationListener接口，当服务列表变化时，就会通知我们。

3.3、总结

Nacos的服务发现分为两种模式：

  • 主动拉取模式。消费者定期主动从Nacos拉取服务列表并缓存起来，在调用服务时优先读取本地缓存中的服务列表。
  • 订阅模式。消费者订阅服务列表，并基于UDP协议来接收服务变更通知。服务列表更新时会发送UDP广播给所有订阅者。

与Eureka相比，Nacos服务状态更新更及时，同步更新更快，避免了因同步慢造成的一系列问题。
