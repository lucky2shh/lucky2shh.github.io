Redis系列-Redis集群搭建

Redis Redis集群
Redis
发布日期:   2023-09-14
更新日期:   2023-08-07
文章字数:   2.4k

1、Redis主从集群

本集群模式为一主二从。

1.1、准备3台服务器

三台服务器信息如下：

IP PORT
192.168.2.23 6379
192.168.2.24 6379
192.168.2.25 6379

1.2、Redis配置

三台服务器的Redis配置相同，这里仅展示一份配置信息。

在redis.conf配置文件中加入如下配置：

# 绑定地址，默认是127.0.0.1，会导致只能在本地访问。修改为0.0.0.0则可以在任意IP访问
bind 0.0.0.0

# 数据库数量，设置为1
databases 1

# 开启RDB
# save ""
save 3600 1
save 300 100
save 60 10000

# 关闭AOF
appendonly no

1.3、启动3台Redis服务

分别在三台服务器中执行如下命令：

redis-server redis.conf

1.4、开启主从关系

将服务器192.168.2.23作为 master节点，剩下2台服务器作为 slave节点。所以只需在slave节点中执行如下命令：

# 192.168.2.24
redis-cli -p 6379
slaveof 192.168.2.23 6379

# 192.168.2.25
redis-cli -p 6379
slaveof 192.168.2.23 6379

执行完上述命令后，若没有什么错误，则表示配置成功。然后可以在master节点中查看集群状态：

# 192.168.2.23
redis-cli -p 6379
info replication

1.5、测试效果

执行如下命令，若一切正常则表示 一主二从集群 搭建成功。

# master节点（192.168.2.23）
set k1 v1
# slave节点（192.168.2.24）：
get k1
set k1 v2
# slave节点（192.168.2.25）：
get k1
set k1 v2

2、Redis哨兵集群

2.1、准备三台服务器

这里以Redis集群服务器为例。

2.2、配置

三台服务器的Redis配置相同，这里仅展示一份配置信息。

在sentinel.conf配置文件中修改如下配置：

port 27001
sentinel monitor mymaster 192.168.2.4 6379 2
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 60000
dir "/opt/redis/data"

2.3、启动Redis服务

分别在三台服务器中执行如下命令：

redis-sentinel sentinel.conf

2.4、测试

关闭master节点服务，查看两个slave节点的状态，其中有一个会升级为master节点。

2.5、SpringBoot集成

2.5.1、配置

application.yml文件：

spring:
  redis:
    sentinel:
      master: mymaster
      nodes:
        - 192.168.2.4:27001
        - 192.168.2.4:27002
        - 192.168.2.4:27003

配置类：

/**
 * RedisTemplate哨兵模式 - 配置主从读写分离
 *
 * ReadFrom是配置Redis的读取策略，可选值如下：
 *      - MASTER                从master节点读取
 *      - MASTER_PREFERRED      优先读取master节点，master不可用再读取replica
 *      - REPLICA               从slave节点读取
 *      - REPLICA_PREFERRED     优先读取slave节点，slave不可用再读取master
 *
 * @return
*/
@Bean
public LettuceClientConfigurationBuilderCustomizer configurationBuilderCustomizer(){
    return config -> config.readFrom(ReadFrom.REPLICA_PREFERRED);
}

3、Redis分片集群

3.1、准备服务器

在同一台虚拟机中开启6个redis实例，模拟分片集群，信息如下：

IP PORT 角色
192.168.150.101 7001 master
192.168.150.101 7002 master
192.168.150.101 7003 master
192.168.150.101 8001 slave
192.168.150.101 8002 slave
192.168.150.101 8003 slave

3.2、实例创建与配置

创建7001、7002、7003、8001、8002、8003目录：

# 进入/tmp目录
cd /tmp
# 删除旧的，避免配置干扰
rm -rf 7001 7002 7003
# 创建目录
mkdir 7001 7002 7003 8001 8002 8003

在/tmp下准备一个新的redis.conf文件，内容如下：

port 6379
# 开启集群功能
cluster-enabled yes
# 集群的配置文件名称，不需要我们创建，由redis自己维护
cluster-config-file /tmp/6379/nodes.conf
# 节点心跳失败的超时时间
cluster-node-timeout 5000
# 持久化文件存放目录
dir /tmp/6379
# 绑定地址
bind 0.0.0.0
# 让redis后台运行
daemonize yes
# 注册的实例ip
replica-announce-ip 192.168.2.21
# 保护模式
protected-mode no
# 数据库数量
databases 1
# 日志
logfile /tmp/6379/run.log

将这个文件拷贝到每个目录下：

# 进入/tmp目录
cd /tmp
# 执行拷贝
echo 7001 7002 7003 8001 8002 8003 | xargs -t -n 1 cp redis.conf

修改每个目录下的redis.conf，将其中的6379修改为与所在目录一致：

# 进入/tmp目录
cd /tmp
# 修改配置文件
printf '%s\n' 7001 7002 7003 8001 8002 8003 | xargs -I{} -t sed -i 's/6379/{}/g' {}/redis.conf

3.3、启动服务

因为已经配置了后台启动模式，所以可以直接启动服务：

# 进入/tmp目录
cd /tmp
# 一键启动所有服务
printf '%s\n' 7001 7002 7003 8001 8002 8003 | xargs -I{} -t redis-server {}/redis.conf

通过ps查看状态：

ps -ef | grep redis

如果要关闭所有进程，可以执行命令：

ps -ef | grep redis | awk '{print $2}' | xargs kill

或者（推荐这种方式）：

printf '%s\n' 7001 7002 7003 8001 8002 8003 | xargs -I{} -t redis-cli -p {} shutdown

3.4、创建集群

虽然服务启动了，但是目前每个服务之间都是独立的，没有任何关联。

我们需要执行命令来创建集群，在Redis5.0之前创建集群比较麻烦，5.0之后集群管理命令都集成到了redis-cli中。

3.4.1、Redis5.0前

Redis5.0之前集群命令都是用redis安装包下的src/redis-trib.rb来实现的。因为redis-trib.rb是有ruby语言编写的所以需要安装ruby环境。

# 安装依赖
yum -y install zlib ruby rubygems
gem install redis

然后通过命令来管理集群：

# 进入redis的src目录
cd /tmp/redis-6.2.4/src
# 创建集群
./redis-trib.rb create --replicas 1 192.168.150.101:7001 192.168.150.101:7002 192.168.150.101:7003 192.168.150.101:8001 192.168.150.101:8002 192.168.150.101:8003

3.4.2、Redis5.0后

我们使用的是Redis6.2.4版本，集群管理已经集成到了redis-cli中，格式如下：

redis-cli --cluster create --cluster-replicas 1 192.168.2.21:7001 192.168.2.21:7002 192.168.2.21:7003 192.168.2.21:8001 192.168.2.21:8002 192.168.2.21:8003

命令说明：

  • redis-cli --cluster或者./redis-trib.rb：代表集群操作命令
  • create：代表是创建集群
  • --replicas 1或者--cluster-replicas 1 ：指定集群中每个master的副本个数为1，此时节点总数 ÷ (replicas + 1) 得到的就是master的数量。因此节点列表中的前n个就是master，其它节点都是slave节点，随机分配到不同master

通过命令可以查看集群状态：

redis-cli -p 7001 cluster nodes

3.5、测试

集群操作时，需要给redis-cli加上-c参数才可以：

redis-cli -c -p 7001
set num 123
get num
set a 1

测试成功。

3.7、扩容缩容

3.7.1、扩容

在原有7001、7002、7003、8001、8002、8003的基础上，增加一个7004节点。

3.7.1.1、实例准备与配置

cd /tmp
mkdir 7004
cp redis.conf 7004

修改配置，将其中的6379改为7004：

sed -i s/6379/7004/g 7004/redis.conf

3.7.1.2、启动7004节点

redis-server 7004/redis.conf

3.7.1.3、将7004节点加入集群

redis-cli --cluster add-node 192.168.2.23:7004 192.168.2.23:7001

查看集群状态：

redis-cli -p 7001 cluster nodes

查看输出信息，发现7004节点中没有任何插槽，接下来要将其他节点中的插槽分给7004节点一部分。

3.7.1.4、为7004节点分配插槽

执行如下命令来分配插槽：

redis-cli --cluster reshard 192.168.2.23:7001

上述命令执行过程中会提示四次，每次都需要输入：

  • 第一个值：表示需要移动多少的槽位？该值最好是(16384 / master节点数量)计算结果。
  • 第二个值：将这些槽位分配给哪个节点，这里填写要分配给的节点ID。节点ID可通过查看集群状态信息来获得。（查看命令： redis-cli -p 7001 cluster nodes）
  • 第三个值：从哪个节点抽取插槽来分配要分配的节点，这里以7001节点为例，填写7001节点ID即可。
  • 第四个值：填写done，然后回车即可。

等待插槽分配完毕后，通过如下命令查看集群状态：

redis-cli -p 7001 cluster nodes

通过输出信息，发现7004节点中已经拥有了刚刚分配的插槽。

3.7.2、缩容

3.7.2.1、迁移7004节点槽位到7001节点

redis-cli --cluster reshard 192.168.2.23:7004

上述命令执行期间，需要输入五个值：

  • How many slots do you want to move (from 1 to 16384)?
    需要移动的槽位数量，这些槽位来自 节点7004。
  • What is the receiving node ID?
    将移动的槽位分配给哪个节点，这里填写节点ID。比如分配给节点7001，那么就写节点7001的ID值。
  • Source node #1:
    从哪个节点中移动槽位。比如填写节点7004的ID值，那么就表示从节点7004中移动槽位到其它节点。
  • Source node #2:
    输入done即可。
  • Do you want to proceed with the proposed reshard plan (yes/no)?
    输入yes即可。

3.7.2.2、删除7004节点

redis-cli --cluster del-node 192.168.2.23:7004 7004节点的ID号

删除7004节点后，通过如下命令查看集群状态：

redis-cli -p 7001 cluster nodes

通过输出信息，发现7004节点已经不在集群，证明删除成功。

3.8、故障转移

3.8.1、场景描述

假如 master节点6381机器 宕机，那么跟随其的 slave节点6386 是否会升级为master节点呢？下面一一验证。

3.8.2、模拟宕机

这里通过 关闭master节点6381 来模拟真实的宕机。

3.8.3、节点升级

master节点6381 宕机后，跟随其的slave节点8386会升级为master节点。这个升级过程是自动完成的。

3.8.4、宕机恢复

master节点宕机后再次恢复时只会作为slave节点 跟随 master节点6386。

详情信息可通过 集群查看命令 来查看。

3.9、SpringBoot集成

3.9.1、配置

application.yml文件：

spring:
  redis:
    cluster:
      nodes:
        - 192.168.2.4:7001
        - 192.168.2.4:7002
        - 192.168.2.4:7003
        - 192.168.2.4:7004
        - 192.168.2.4:7005
        - 192.168.2.4:7006

配置类：

@Bean
public LettuceClientConfigurationBuilderCustomizer configurationBuilderCustomizer(){
    return config -> config.readFrom(ReadFrom.REPLICA_PREFERRED);
}
