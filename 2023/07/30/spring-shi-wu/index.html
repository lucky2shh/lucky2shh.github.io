Spring系列-Spring事务

Spring Spring事务
Spring
发布日期:   2023-07-30
文章字数:   1.4k

基础知识

1.1、事务定义

事务指的是 通过一系列操作来完成某个功能，这些操作要么全成功，要么全失败。这个保证正确完成某个功能的规则称为事务。

1.2、事务控制方式

Spring事务提供了两种事务控制方式：

  • 编程式事务

    使用事务代码控制业务代码 的正确执行，事务代码和业务代码耦合在一起。这种方式不常用。

  • 声明式事务

    Spring将事务代码进行封装，并通过注解来完成事务控制，实现了事务代码与业务代码的解耦。推荐这种方式。

1.3、事务相关类

  • PlatformTransactionManager（平台事务管理器）

    一个标准接口，具备事务提交、回滚、获取事务对象等功能。不同框架有不同的实现。

  • TransactionDefinition（事务定义）

    封装事务的隔离级别、传播行为、过期时间等信息。

  • TransactionStatus（事务状态）

    存储事务的状态信息。如事务是否提交、回滚等。

1.4、事务特性

  • 原子性（atomicity）：强调事务不可分割.。
  • 一致性（consistency）：事务执行前后，数据的完整性保持一致.。
  • 隔离性（isolation）：一个事务执行过程中，不应该受到其他事务的干扰。
  • 持久性（durability）：事务一旦结束，数据就持久到数据库。

1.5、事务传播行为（propagation）

PROPAGATION_*（REQUIRED、SUPPORTS…）

  1. REQUIRED（默认）：A调用B，B需要事务，如果A有事务，那么B就加入到A的事务，否则B单独创建一个事务。
  2. REQUIRES_NEW：A调用B，B需要一个新事务，如果A有事务那么就挂起，B单独创建一个新事务。
  3. SUPPORTS：A调用B，A有事务那么B就加入该事务，否则B就以非事务方式执行。
  4. NOT_SUPPORTED：A调用B，B以非事务方式执行，A有事务就挂起。
  5. MANDATORY：A调用B，B要加入A的事务，A没有事务就抛异常。
  6. NEVER：A调用B，B以非事务方式执行，A有事务则抛出异常。
  7. NESTED：A调用B，B新建一个事务，A有事务则以嵌套事务存在，否则就以新的事务执行。

1.6、事务隔离级别（isolation）

事务隔离级别主要用于解决如下问题：

  • 脏读：一个事务读到了另一个事务未提交的数据。
  • 不可重复读：一个事务读到了另一个事务已经提交的 update 数据，导致多次查询结果不一致.。
  • 幻读：一个事务读到了另一个事务已经提交的 insert 数据，导致多次查询结果不一致。

事务隔离级别有如下几种：

  1. DEFAULT：默认的隔离级别，数据库默认的事务隔离级别；（MySQL默认为 可重复读；Oracle默认为 读已提交）。
  2. Read Uncommitted（读未提交）：一个事务 可以读取 未提交的事务的数据。会发生脏读 、不可重复读、幻读问题。
  3. Read Committed（读已提交）：一个事务 可以读取 已提交的事务的数据。解决了脏读，但没有解决不可重复读、幻读问题。
  4. Repeatable Read（可重复读）：可以防止脏读、不可重复读，但幻读问题依然存在。
  5. Serializable（串行化）：代价高但可靠的一种隔离级别。事务顺序执行。可防止脏读、不可重复读、幻读。

MySQL支持全部4种隔离级别，默认隔离级别是可重复读。

Oracle仅支持读已提交、串行化 两种隔离级别，默认是读已提交。

1.7、其它属性

  • read-only：是否只读；
  • timeout：超时时间；

使用事务

2.1、环境准备

要使用Spring的事务，需要导入spring-jdbc依赖。其它依赖可根据实际开发环境自行选择。

2.2、XML声明式事务

xml配置文件：

<bean id="transactionManager" class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
    <property name="dataSource" ref="dataSource"></property>
</bean>

<tx:advice id="txAdvice" transaction-manager="transactionManager">
    <tx:attributes>
        <tx:method name="*" isolation="REPEATABLE_READ" propagation="REQUIRED"/>
    </tx:attributes>
</tx:advice>

<aop:config>
    <aop:pointcut id="txPointcut" expression="execution(* cn.shh.demo.spring.aop.transaction.*.*(..))"/>
    <aop:advisor advice-ref="txAdvice" pointcut-ref="txPointcut"/>
</aop:config>

只需以上xml配置，就可以实现事务控制。

2.3、注解声明式事务

2.3.1、方式一

xml配置文件：

<bean id="transactionManager" class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
    <property name="dataSource" ref="dataSource"></property>
</bean>

<tx:annotation-driven transaction-manager="transactionManager"/>

xml配置完成后，在需要事务控制的业务方法上标注注解 @Transactional即可实现事务控制。

2.3.2、方式二

java配置类：

@ComponentScan("cn.shh.demo.spring.aop.transaction")
@Configuration
@EnableTransactionManagement
public class TransactionConfig {
    @Bean
    public DataSource dataSource(){
        DruidDataSource dataSource = new DruidDataSource();
        String fix = "?characterEncoding=utf8&useSSL=false&serverTimezone=GMT%2B8";
        dataSource.setUrl("jdbc:mysql://localhost:3307/mybatis_plus" + fix);
        dataSource.setDriverClassName("com.mysql.cj.jdbc.Driver");
        dataSource.setUsername("root");
        dataSource.setPassword("root666");
        return dataSource;
    }
    @Bean
    public JdbcTemplate jdbcTemplate(DataSource dataSource){
        return new JdbcTemplate(dataSource);
    }
    @Bean
    public DataSourceTransactionManager dataSourceTransactionManager(DataSource dataSource){
        return new DataSourceTransactionManager(dataSource);
    }
}

然后在需要事务控制的业务方法上标注注解 @Transactional即可实现事务控制。

原理解析

1.1、入口 tx:advice

（1）向容器中注入TransactionInterceptor

通过标签解析器 TxAdviceBeanDefinitionParser 向容器中注入 TransactionInterceptor（名字为txAdvice）。

（2）使用TransactionInterceptor拦截方法，开启、提交/回滚事务

TransactionInterceptor 实现了AOP的 MethodInterceptor，并重写invoke方法，invoke就是环绕通知。环绕前开启事务，环绕后提交/回滚事务。
