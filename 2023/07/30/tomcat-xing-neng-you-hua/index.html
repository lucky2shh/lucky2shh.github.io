Tomcat系列-Tomcat优化

Tomcat优化

1、内存优化

启动时告诉JVM需要一块大内存（调优内存是最直接的方式）。

修改配置文件catalina.sh（Windows下为catalina.bat，Linux下为catalina.sh）：

JAVA_OPTS='-Xms256m -Xmx512m'
  • -Xms<size> ：JVM 初始化堆的大小；
  • -Xmx<size>：JVM 堆的最大值；

2、线程优化

2.1、连接线程优化

在 server.xml 中进行如下配置：

<Connector port="80" protocol="HTTP/1.1"  
    connectionTimeout="20000" 
    redirectPort="8443"   
    executor="tomcatThreadPool"  
    enableLookups="false"   
    maxIdleTime= "60000"
    acceptCount="100"   
    maxPostSize="10485760" 
    acceptorThreadCount="2"    
    disableUploadTimeout="true"   
    URIEncoding="utf-8"
    keepAliveTimeout ="6000"  
    maxKeppAliveRequests="500" />
  • port：连接端口；
  • protocol：连接器使用的传输方式；
  • executor： 连接器使用的线程池名称；
  • enableLookups：禁用 DNS查询；
  • maxIdleTime：线程空闲时间，超过该时间后，空闲线程会被销毁；（默认值为6000（1分钟），单位毫秒。）
  • acceptCount：当没有处理请求的空闲线程时，可以放到队列中的请求数量，超过该阈值的请求将不予处理；（默认设置 100 。）
  • maxPostSize：限制 FORM URL参数方式的POST请求的内容大小；（单位字节，默认是 2097152(2兆)，10485760 为 10M。如果要禁用限制，则可以设置为 -1。）
  • acceptorThreadCount： 接收连接的线程数量；（默认值是1，如果是多核 CPU，则一般配置为 2）
  • disableUploadTimeOut：允许Servlet容器使用一个较长的连接超时值来完成复杂的逻辑；（默认值为false）
  • keepAliveTimeout：连接持续存活时间；
  • maxKeepAliveRequests：连接能承受的最大请求数量，超过该阈值的连接将被关闭；（该参数值为1时表示禁用，-1表示不限制个数，默认100个。一般设置在100~200之间。）

2.2、业务处理线程优化

在server.xml中进行如下编辑：

<Executor name="tomcatThreadPool" namePrefix="catalina-exec-"
    maxThreads="150" 
    minSpareThreads="100"
    prestartminSpareThreads="true"
    maxQueueSize="100"/>
  • name：线程名称；
  • namePrefix：线程前缀；
  • maxThreads：最大并发连接数；（默认200，一般设置为500~ 800 ，要根据实际情况来定）
  • minSpareThreads：Tomcat启动初始化的线程数；（默认值25）
  • prestartminSpareThreads：tomcat初始化时会初始化参数minSpareThreads值，不设置就不会初始化；
  • maxQueueSize：最大的等待队列数，超过则拒绝请求；

相关参数解析如下：

  1. maxThreads：客户请求最大线程数；
  2. minSpareThreads：Tomcat初始化时创建的 socket 线程数；
  3. maxSpareThreads：Tomcat连接器的最大空闲 socket 线程数；
  4. enableLookups：若设为true, 则支持域名解析，可把 ip 地址解析为主机名；
  5. redirectPort：在需要基于安全通道的场合，把客户请求转发到基于SSL 的 redirectPort 端口；
  6. acceptAccount：监听端口队列最大数，满了之后客户请求会被拒绝（不能小于maxSpareThreads ）；
  7. connectionTimeout：连接超时；
  8. minProcessors：服务器创建时的最小处理线程数；
  9. maxProcessors：服务器同时最大处理线程数；
  10. URIEncoding：URL统一编码；

3、IO优化

IO类型主要有如下三种：

  1. 同步阻塞（BIO）：一个线程一个连接，当然可以通过线程池机制改善；

    适用场景：BIO适用于连接数少且固定的场景。这种方式对服务器资源要求比较高，并发局限于应用中，JDK1.4以前的唯一选择，但程序直观简单易理解。

  2. 异步阻塞（NIO）：同一个线程处理多个连接(多路复用)；

    适用场景：NIO适用于连接数多且连接较短的场景。比如聊天服务器，并发局限于应用中，编程比较复杂，JDK1.4开始支持。

  3. 异步非阻塞（NIO2，又叫AIO）：与NIO的区别主要在操作系统底层；

    适用场景：AIO适用于 连接数多且连接较长 的场景。比如相册服务器，充分调用OS参与并发操作，编程比较复杂，JDK7开始支持。

如下配置可实现 Tomcat IO 模型的切换：

<Connector port="80" protocol="org.apache.coyote.http11.Http11NioProtocol" 
           connectionTimeout="20000" 
           URIEncoding="UTF-8" 
           useBodyEncodingForURI="true" 
           enableLookups="false" 
           redirectPort="8443" />

Connector使用哪种protocol，可以通过<connector>元素中的protocol属性进行指定，也可以使用默认值。

指定的protocol取值及对应的协议如下：

  • HTTP/1.1：默认值，使用的协议与Tomcat版本有关
  • org.apache.coyote.http11.Http11Protocol：BIO
  • org.apache.coyote.http11.Http11NioProtocol：NIO
  • org.apache.coyote.http11.Http11Nio2Protocol：NIO2
  • org.apache.coyote.http11.Http11AprProtocol：APR

如果没有指定protocol，则使用默认值HTTP/1.1，其含义如下：在Tomcat7中，自动选取使用BIO或APR（如果找到APR需要的本地库，则使用APR，否则使用BIO）；在Tomcat8中，自动选取使用NIO或APR（如果找到APR需要的本地库，则使用APR，否则使用NIO）

4、缓存压缩优化

4.1、压缩优化定义

压缩优化是指：将返回的 html 页面等内容压缩成 gzip 格式后发送给浏览器，浏览器在本地解压缩的过程。

4.2、开启压缩优化

可以根据自己的业务场景在 server.xml 中进行如下相关配置：

<Connector port="9027"
           protocol="HTTP/1.1"
           maxHttpHeaderSize="8192"
           maxThreads="1000"
           minSpareThreads="100"
           maxSpareThreads="1000"
           minProcessors="100"
           maxProcessors="1000"
           enableLookups="false"
           compression="on"
           compressionMinSize="2048"
           compressableMimeType="text/html,text/xml,text/javascript,text/css,text/plain"
           connectionTimeout="20000"
           URIEncoding="utf-8"
           acceptCount="1000"
           redirectPort="8443"
           disableUploadTimeout="true"/>

参数解析：

  • **compression=”on”**：打开压缩功能；
  • **compressionMinSize=”2048”**：启用压缩的输出内容大小，默认为2KB；
  • **noCompressionUserAgents=”gozilla,traviata”**：对于以下的浏览器，不启用压缩；
  • **compressableMimeType=”text/html,text/xml,text/javascript,text/css,text/plain”**：哪些资源类型需要压缩；

4.3、缓存策略

开启浏览器缓存，可以快速读取webapps文件夹里的静态内容，提高了整体性能。

  1. 浏览器第一次请求tomcat服务器某资源。
  2. tomcat拿到资源后将该资源最后修改的时间保存在响应头的Last-Modified中 （Last-Modified: Tue, 11 Jul 2000 18:23:51 GMT）。
  3. 浏览器第二次访问tomcat请求该资源时将该资源的最后修改时间放入If-Modified-Since请求头中。
  4. 服务器收到请求并比较请求中的If-Modified-Since与资源最后修改时间 Last-Modified 是否一致，如果一致则发送响应代码304，告知其在本地缓存拿就可以了。

5、安全优化

安全优化参考如下：

  1. 降权启动

  2. 修改端口号

  3. 更改 tomcat 关闭指令

  4. 修改管理员用户名、密码

  5. 清空站点目录root下的管理页面等

  6. 错误页面的优雅展示

  7. 隐藏版本号

  8. 禁用DNS查询

    当web应用记录客户端信息时，它也会记录客户端IP地址或通过域名服务器查找机器名然后转为IP地址。DNS查询需要网络开销，且还有远距离获取IP的开销。

    关闭DNS查询可提高性能，关闭DNS查询的方式是：将 server.xml 文件中参数 enableLookups 的值设为false即可。

  9. 设置session过期时间
    Tomcat采用数据库连接池技术，该技术的执行逻辑是：当用户在指设置的 session 超时时间内没有对数据库执行操作时，就会自动关闭该连接，避免浪费宝贵的数据库连接资源。设置 session 过期时间的方法是 在 tomcat_home / conf / web.xml文件中找到 session-config 标签的子标签 session-timeout 标签，为该子标签设置超时时间即可，单位是分钟。

  10. server.xml中进行如下定义：

<Context path="/test" docBase="/test" 
         isWARExpanded="true"
         isWARValidated="false" 
         isInvokerEnabled="true"
         isWorkDirPersistent="false" 
         defaultSessionTimeOut="3600" />

  1. 项目所在目录下的web.xml中定义：

    <session-config>
　　<session-timeout>20</session-timeout>
</session-config>

  2. 在程序中定义：

    session.setMaxInactiveInterval(30*60);   // 单位：秒，设为-1表示永不过期。

6、APR（大杀器）

6.1、概述

APR（Apache Portable Runtime）是Apache的一个库，它提供了跨平台的API，用于支持Apache HTTP Server 及其它Apache项目。APR通过对操作系统底层功能的封装，提供了一种高效、可移植和可靠的编程接口。

Tomcat 中 APR 通常指的是 APR/native 组件，它是一个与操作系统级别的库进行交互的接口。APR/native 组件可以与Tomcat一起使用，以提高性能和可扩展性。

APR/native 组件的主要功能包括：

  • 提供高效的网络IO操作：通过操作系统级别的多路复用技术，可以显著提高网络性能。
  • 支持更高级的加密功能：可以利用操作系统提供的高级加密库，提供更好的加密性和安全性。
  • 提供更好的静态资源服务功能：利用操作系统的高性能文件IO接口，提供高效的静态资源服务能力。

6.2、安装使用

安装使用步骤参考如下：

  1. 从Apache官网下载APR的库。

  2. 编译安装APR库

    ./configure
make
make install

  3. 编译安装APR/native模块

    ./configure --with-apr=/path/to/apr/source
make
make install

    /path/to/apr/source是APR库源代码的路径。

  4. 配置Tomcat使用APR/native模块

    打开 conf/server.xml 文件，在该文件中找到标签 Connector，在该标签中配置属性 protocol=org.apache.coyote.http11.Http11AprProtocol，修改完后保存server.xml文件。

  5. 重启Tomcat服务使其生效

Tomcat之JVM优化

在catalina.sh 文件中加入如下配置：

JAVA_OPTS="-Xms2G -Xmx2G -Xmn512m -XX:MetaspaceSize=512M -XX:MaxMetaspaceSize=512M -XX:+UseConcMarkSweepGC -XX:+CMSClassUnloadingEnabled -XX:CMSInitiatingOccupancyFraction=75 -XX:+UseCMSInitiatingOccupancyOnly -XX:+HeapDumpOnOutOfMemoryError -verbose:gc -XX:+PrintGCDetails -XX:+PrintGCTimeStamps -XX:+PrintGCDateStamps -Xloggc:/appl/gc.log"

参数解析：

  • -Xms2G：初始堆内存；
  • -Xmx2G：最大堆内存；
  • -Xmn512m：年轻代大小；
  • -XX:MetaspaceSize=512M：初始元空间大小，达到该值会触发垃圾回收并进行类型卸载，同时GC会对该值进行动态调整：如果垃圾回收后释放了大量空间就降低该值；如果释放了少量空间，那么就在不超过 MaxMetaspaceSize设定值的前提下适当提高该值。
  • -XX:+UseConcMarkSweepGC：使用 并发标记清除类型的 CMS垃圾回收器；
  • -XX:+CMSClassUnloadingEnabled： 表示 是否启用类卸载；（+表示启用，-表示关闭，默认关闭。该功能只有在使用CM回收器时才有效。）
  • -XX:+HeapDumpOnOutOfMemoryError：表示 发生OOM时会自动生成DUMP文件；
  • -XX:HeapDumpPath=${目录}：表示生成DUMP文件的路径，也可以指定文件名称，例如：-XX:HeapDumpPath=${目录}/java_heapdump.hprof。如果不指定文件名，默认为：java_<pid>_<date>_<time>_heapDump.hprof；
  • -verbose:gc：输出GC日志 ， -XX:+PrintGC 与 -verbose:gc 是一样的，可以认为-verbose:gc 是 -XX:+PrintGC的别名；
  • -XX:+PrintGCDetails：打印GC详细信息；
  • -XX:+PrintGCTimeStamps：输出GC时间戳；（以JVM启动到当期的总时长的时间戳形式）
  • -XX:+PrintGCDateStamps：输出GC的时间戳；（以日期的形式，如 2013-05-04T21:53:59.234+0800）
  • -Xloggc:/appl/gc.log：定义gc日志目录；
  • -XX:CMSInitiatingOccupancyFraction=75：内存占用率达到75%时开启CMS GC；
  • -XX:+UseCMSInitiatingOccupancyOnly：设定该选项 表示 CMS始终使用参数<font color='orange'>-XX:CMSInitiatingOccupancyFraction指定的垃圾回收阈值，如果不设定该选项，则表示只有在最开始阶段使用上面参数指定的垃圾回收阈值，后期会根据实际情况动态调整垃圾回收阈值；

操作系统优化（Linux）

1、最大打开文件数

1.1、系统级句柄数量

  • 查看

    cat /proc/sys/fs/file-max

  • 修改

    echo 数量 > /proc/sys/fs/file-max

  • 永久生效

    # 1、修改文件
vim /etc/sysctl.conf
# 2、在文件末尾添加如下内容
fs.file-max = 2000000
# 3、执行如下命令，使修改配置立即生效
sysctl -p

1.2、进程级句柄数量

# 1、打开文件
vim /etc/security/limits.conf
# 2、在文件最后添加如下内容
* soft nproc 100001     # 警告设定 所有用户最大打开进程数为10001
* hard nproc 100001     # 严格设定 所有用户最大打开进程数为100001
* soft nofile 100001    # 警告设定 所有用户最大打开文件数为100001
* hard nofile 100002    # 严格设定 所有用户最大打开文件数为100002
  • soft 表示警告设定，超过该值会有警告信息；
  • hard 表示严格设定，不能超过该值；

注意：修改后，要重启服务器才生效。

2、TCP参数优化

以下命令参数的修改全部基于/etc/sysctl.conf文件，修改完后，通过命令sysctl -p使其修改生效。

#（4G 内存机器 使用，TCP连接最多约使用1.6GB内存 ， 393216*4096/1024/1024=1.6G）
#内核分配给TCP连接的内存，单位是Page，1 Page = 4096 Bytes
net.ipv4.tcp_mem = 196608  262144  393216  

#（8G 内存使用，TCP连接最多约使用4GB内存）     
net.ipv4.tcp_mem = 524288  699050  1048576      

#为每个TCP连接分配的读、写缓冲区内存大小，单位是Byte
# 一般按照缺省值分配，上面的例子就是读写均为8KB，共16KB
#1.6G 内存服务器， TCP内存能容纳的连接数，约为  1600MB/16KB = 100K = 10万
#4.G TCP内存能容纳的连接数，约为  4000MB/16KB = 250K = 25万
#                  最小内存  缺省内存  最大内存
net.ipv4.tcp_rmem = 4096     8192    4194304
net.ipv4.tcp_wmem = 4096     8192    4194304

#（端口最大的监听队列长度）
#同时，修改下全局配置
# echo 4000 > /proc/sys/net/core/somaxconn 定义了系统中每一个端口最大的监听队列的长度,这是个全局的参数,默认值为128,
net.core.somaxconn= 4000

#表示开启SYN Cookies。当出现SYN等待队列溢出时，启用cookies来处理，可防范少量SYN攻击，默认为0，表示关闭；
net.ipv4.tcp_syncookies = 1

#表示开启重用。允许将TIME-WAIT sockets重新用于新的TCP连接，默认为0，表示关闭；
net.ipv4.tcp_tw_reuse = 1

#表示开启TCP连接中TIME-WAIT sockets的快速回收，默认为0，表示关闭；
net.ipv4.tcp_tw_recycle = 1

#修改系統默认的 TIMEOUT 时间。
net.ipv4.tcp_fin_timeout = 30

#表示当keepalive起作用时，TCP发送keepalive消息的频度。缺省是2小时，改为20分钟。
net.ipv4.tcp_keepalive_time = 1200  

#表示用于向外连接的端口范围。缺省情况下很小：32768到61000，改为10000到65000。
#（注意：不要将最低值设太低，否则可能会占用正常端口！）
net.ipv4.ip_local_port_range = 10000 65000  

#表示SYN队列长度，默认为1024，加大队列长度为8192，可以容纳更多等待连接的网络连接数。
net.ipv4.tcp_max_syn_backlog = 8192

#表示系统同时保持TIME_WAIT的最大数量，如果超过这个数字，TIME_WAIT将立刻被清除并打印警告信息。默认为180000，改为5000。
net.ipv4.tcp_max_tw_buckets = 5000

#当orphans达到32768个时，会报Out of socket memory，此时占用内存 32K*64KB=2048MB=2GB
#（每个孤儿socket可占用多达64KB内存），实际可能小一些
net.ipv4.tcp_max_orphans = 65536

#孤儿socket废弃前重试的次数，重负载web服务器建议调小，设置较小的数值，可以有效降低orphans的数量
net.ipv4.tcp_orphan_retries = 1

#活动TCP连接重传次数，超过次数视为掉线，放弃连接。缺省值：15，建议设为 2或者3.
net.ipv4.tcp_retries=2

#TCP三次握手的syn/ack阶段，重试次数，缺省5，设为2-3
net.ipv4.tcp_synack_retries=2

# 网络设备的收发包的队列大小
net.core.netdev_max_backlog = 2048
